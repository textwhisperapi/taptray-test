12: Technical Field
13: The invention relates to computer-implemented learning and rehearsal systems, and more particularly to non-destructive transformation of source text into progressive memory cues for recall training and performance preparation.
14: Background
15: Digital lyric/script rehearsal typically relies on static full-text reading or fully manual redaction. Existing approaches frequently suffer from one or more of:
16: 1. destructive modification of source text during memorization workflows;
17: 2. inability to preserve semantic structure (titles, headings, speaker cues) while reducing cues;
18: 3. poor continuity across device states (online/offline);
19: 4. inconsistent rendering and line wrapping when cue density changes; and
20: 5. accidental persistence of transformed practice text as canonical source text.
21: These issues increase rehearsal friction and reduce confidence during preparation and stage execution.
22: Summary of the Invention
23: The invention provides a non-destructive rehearsal cue engine that maintains canonical source text in a first representation and generates a transformed rehearsal representation in a second representation. A user-selectable cue level controls token truncation, while configurable line-preservation rules protect semantically important lines from truncation. The transformed view is rendered for practice without altering canonical content. Persistence safeguards prevent saving transformed text as source while transformation mode is active.
24: In embodiments, punctuation and emoji are preserved, and transformed rendering can be regenerated after online/offline load. The result is deterministic, repeatable cue progression with source integrity.
25: The invention is principally directed to non-destructive trim-based cue transformation; cache and reload behavior are optional implementation embodiments and not required for core operation.

32: Definitions
33: 1. Canonical representation: authoritative stored text source used for save/load operations.
34: 2. Transformed rehearsal representation: derived cue text used for practice display.
35: 3. Hint level: user-controlled truncation parameter.
36: 4. Line class: a categorized text line (for example title, hashtag, heading, speaker line) used for preservation rules.
37: 5. Non-destructive mode: operation where transformed text does not overwrite canonical content.
38: Detailed Description
39: 1. System Overview
40: The system includes:
41: 1. a canonical text source buffer;
42: 2. a transformation engine;
43: 3. a transformed text output buffer;
44: 4. a user interface controller including hint-level input;
45: 5. a persistence controller enforcing non-destructive save constraints; and
46: 6. optional cache coordination for offline/online regeneration.
47: Canonical text is maintained in an unchanged source representation. On hint-level updates, the transformation engine deterministically regenerates transformed rehearsal text from canonical input.
48: 2. Canonical and Transformed Representations
49: In one embodiment, the canonical representation is maintained in a mirrored source pane or hidden canonical buffer. The transformed output is displayed in a practice pane. The transformed pane may be editable for interaction but is not accepted as canonical save content while transformation is active.
50: 3. Line Classification and Preservation
51: Before token transformation, lines are classified. Preserved classes can include:
52: 1. first non-empty/title line;
53: 2. hashtag-marked lines;
54: 3. all-caps heading lines; and
55: 4. speaker-identifier lines.
56: Preserved lines are rendered with minimal or no truncation, while non-preserved lines are transformed according to hint level.
57: 4. Token-Level Cue Transformation
58: For non-preserved lines, each token is transformed based on hint level.
59: 1. alphanumeric components are truncated to a retained prefix length;
60: 2. punctuation marks are preserved;
61: 3. emoji symbols are preserved;
62: 4. transformed tokens are concatenated for output rendering.
63: This produces progressive cues that maintain rhythm/shape/context while reducing direct lexical exposure.
64: 5. Persistence Safeguards
65: A save guard prevents transformed text from being persisted as canonical when hint level is non-zero (or transformation mode is active). To store canonical edits, the interface requires reset to full-source state. This preserves source integrity and prevents accidental corruption.
66: 6. Offline/Online Regeneration
67: When content is loaded from cache or server, canonical representation is restored first. The transformed representation is then regenerated from canonical using the active hint level. This ensures deterministic output regardless of connectivity state.
68: This regeneration path is an optional embodiment and not required for the core trim-based non-destructive transformation method.
69: 7. Example Workflow
70: 1. Load canonical rehearsal text.
71: 2. User sets hint level > 0.
72: 3. System classifies lines and preserves selected classes.
73: 4. Remaining tokens are truncated with punctuation/emoji retention.
74: 5. Output is rendered as transformed rehearsal view.
75: 6. Save is blocked until hint level reset to canonical mode.
76: 8. Technical Effects
77: 1. improved memory-recall progression without source damage;
78: 2. reduced rehearsal friction via deterministic cue generation;
79: 3. consistent cross-session rendering due to canonical regeneration;
80: 4. lower risk of accidental destructive saves.
81: Reference Implementation Examples
82: The following examples are explanatory embodiments that show one way to implement the claimed method and system. They do not limit claim scope.
83: Example A: Cue Generation Pipeline (Pseudocode)
84: function buildTransformedView(canonicalText, hintLevel):
85: lines = splitIntoLines(canonicalText)
86: outputLines = []
87: for line in lines:
88: if isPreservedLineClass(line):
89: outputLines.append(line)
90: continue
91: tokens = tokenize(line)
92: transformedTokens = []
93: for token in tokens:
94: transformedTokens.append(transformToken(token, hintLevel))
95: outputLines.append(joinTokens(transformedTokens))
96: return outputLines
97: What this does: Generates a transformed rehearsal representation from canonical representation by preserving configured line classes and truncating remaining tokens according to hint level.
98: Why this matters: Shows deterministic cue generation that protects semantic structure while reducing direct lexical exposure.
99: Example B: Save Guard Enforcement (Implementation-Style)
100: function canPersistCanonical(hintLevel, transformModeActive) {
101: // Canonical writes are only allowed in full-source mode.
102: return hintLevel === 0 && transformModeActive === false;
103: }
104: function saveCanonical(sourceText, transformedText, hintLevel, transformModeActive) {
105: if (!canPersistCanonical(hintLevel, transformModeActive)) {
106: throw new Error("Save blocked: transformed rehearsal state cannot overwrite canonical text.");
107: }
108: persistCanonicalText(sourceText);
109: }
110: What this does: Blocks canonical persistence whenever transformation mode is active or hint level is not at full-source mode.
111: Why this matters: Implements the non-destructive safeguard that prevents transformed rehearsal text from overwriting canonical source text.
112: Example C: Deterministic Regeneration on Reload (Pseudocode)
113: onLoad(itemId):
114: canonical = loadCanonical(itemId)
115: hintLevel = loadUserHintLevel(itemId)
116: transformed = buildTransformedView(canonical, hintLevel)
117: renderCanonicalAndTransformed(canonical, transformed)
118: What this does: Restores canonical representation first, then regenerates transformed representation from the same canonical input and active hint level.
119: Why this matters: Demonstrates consistent behavior across online/offline reload states.
120: Example Claim Set
121: Independent Claim 1 (Method)
122: 1. A computer-implemented method for rehearsal recall training, comprising:
123: 2. storing canonical text for a rehearsal item in a first representation;
124: 3. generating, from the canonical text, a transformed rehearsal representation by truncating words according to a user-selectable hint level;
125: 4. preserving predefined line classes from truncation, the line classes including at least one of title lines, hashtag-marked lines, and all-caps heading lines;
126: 5. rendering the transformed rehearsal representation in a user-facing text viewport while retaining the canonical text unchanged in a separate source representation; and
127: 6. preventing persistence of transformed text as canonical text while truncation mode is active.

152: Abstract
153: Disclosed is a computer-implemented rehearsal recall system that generates progressive memory cues from canonical text without destructive source modification. Canonical text is retained in a first representation while a transformed rehearsal representation is generated by token truncation according to a user-selected hint level. Line-preservation rules exempt semantically important line classes, including title and heading-like lines, from truncation. The transformed representation is rendered for practice. Persistence safeguards prevent transformed practice text from being saved as canonical text while transformation mode is active. The system supports deterministic regeneration of transformed cues after online or offline load, enabling consistent rehearsal progression and reduced source-corruption risk.
